Motivation: Tool Calling is an Inference Bottleneck

Why This Matters
  • LM agents rely heavily on tools (file search, code execution, web search, APIs). Each tool call breaks generation into fragmented requests.
  • Sequential dependency: model generates → stops → client runs tool → sends result back → engine re-prefills. Latencies accumulate across turns.
  • Eviction overhead: sequences are removed from the active batch during tool execution. KV-cache may be discarded. Re-scheduling and re-prefilling grow with prompt length.
  • Multi-tenant contention: many concurrent agents compete for KV-cache slots; prefix-caching becomes less effective under load.
  • Existing mitigations are limited: parallel tool calls require model support & sequential tools are excluded; prefix caching doesn't eliminate eviction or re-scheduling costs.
Headline Result: Up to 196 tok/s throughput improvement with 32 concurrent gpt-oss-120b agents on vLLM.

Background: The Tool Calling Pipeline

Standard Inference Cycle with Tools
  • Prefill phase: Engine processes all prompt tokens to produce the first output token (compute-bound).
  • Decode phase: Auto-regressive generation of subsequent tokens until a stop or tool-call token (memory-bound).
  • Tool execution: Generation halts; sequence evicted from batch; client runs tool externally (0.1s - minutes).
  • Re-entry: Client submits new request = original prompt + intermediate text + tool output → engine must re-prefill the growing history before decoding again.

Existing Partial Solutions
  • Parallel tool calls: model emits multiple tools in one turn → reduces round-trips, but requires model training & can't handle dependent tools.
  • Prefix caching: reuses KV-cache for repeated prefixes → cuts re-prefill cost, but doesn't eliminate eviction, re-scheduling, or multi-tenant cache pressure.

Key Insight: Speculative Decoding vs. Speculative Tool Calling

Speculative Decoding (Background)
  • A small draft model proposes several tokens ahead of the target model.
  • Target model validates in a single forward pass; accepted tokens are "free".
  • Typically only 3-10 tokens ahead → saves milliseconds per step.
  • Useful for turning memory-bound decode into compute-bound verification.
Limitation for Tool Calling
  • If we launch tools as the draft generates them, we only overlap tool calls with a few generated tokens — milliseconds of savings, dominated by tool latency.
Speculative Tool Calling (This Paper)
  • Instead of speculating individual tokens, speculate the entire tool call (function name + arguments) early.
  • Launch the tool asynchronously while the main model is still generating → overlap tool execution with generation.
  • Potential savings: seconds (tool latency) instead of milliseconds.
  • Key difference: speculate as soon as possible and, for stateless/cheap tools, as many times as possible.

Core idea: Use a small, fast model to predict which tool the main model will call, then pre-execute it to hide latency.

Client-side Speculative Tool Calling

Algorithm Overview
  • Step 1: Send prompt to main model M and speculative model S in parallel.
  • Step 2: S returns a tool prediction (faster, smaller). Launch tool asynchronously and store future in a tool cache.
  • Step 3: M returns. If it called the same tool → cache hit: await the pre-launched future instead of re-executing.
  • Step 4: If cache miss → fall back to normal tool execution.
Key Properties
  • Works with ANY unmodified API — no engine changes needed. Purely client-side.
  • Can run λ speculative samples in parallel to boost hit rate.
  • Only applicable to stateless, cheap tools (web search, file read, etc.).
  • Multiple speculation samples: smaller models + more samples ≈ larger model accuracy.

Client-side Analysis: Speedup Bound

Runtime Model

For N repeated agent turns:

Tstandard = N (G + T)

Tspec = αN · max{G, g+T} + (1-α)N(G+T)
  • G = avg main model generation time
  • g = avg speculative model generation time
  • T = avg tool call duration
  • α = speculative model acceptance rate
Lemma: Sspec < 2
  • Speedup is strictly increasing in α → any α > 0 helps.
  • Maximum speedup approaches 2× as g → 0 but never reaches it.
  • Conditions for speedup: α > 0 and g < G (S is faster than M).
  • Best gains when T ≈ G (tool latency similar to generation time).
Speedup heatmap across acceptance rate α, generation time ratio g/G, and tool latency T. Best speedups at T≈G, α>0.5, g/G<0.5.

Engine-side Speculative Tool Calling

Three Optimizations (O1 + O2 + O3)
  • O1 - Speculate tools: Same as client-side — speculative model predicts tools, client launches them async. Results are posted to the engine's tool cache.
  • O2 - Early exit decoding: When engine detects a tool-call-start token, it looks up the cache. If found, cached tool call tokens are injected as draft tokens and validated via speculative sampling in one forward pass. Avoids generating tool call argument tokens one-by-one.
  • O3 - Avoid eviction: When tool-call-end token is detected and cache has the result, append tool output directly to KV-cache. No eviction, no re-scheduling, no re-prefill of growing history.

Requires engine modifications (implemented in custom vLLM fork), but yields additional savings on top of client-side approach.

Figure: Engine-side approach with three optimizations: speculate tools (O1), early exit via speculative sampling (O2), and avoid eviction (O3).

Engine-side Analysis: Runtime Equations

Vanilla Runtime (K agent turns)
Tvanilla = 2Ko + φ Σ Xi + δ Σ(Ri + ti) + Σ Ti
  • o = API & eviction overhead per turn
  • φ = prefill rate (s/tok)
  • δ = decode rate (s/tok)
  • Ri = reasoning tokens, ti = tool call tokens
  • Ti = tool execution duration
With Prefix Caching
Tcached = 2Ko + φ(X1 + Σ(ti+to,i)) + δΣ(Ri+ti) + ΣTi
With Speculation (α accuracy)
T*spec = (1-α)2Ko + φ(X1 + Σ(ti+to,i))
  + δ(αK + ΣRi + (1-α)Σti) + (1-α)ΣTi
  • At α=1 saves: 2Ko (overhead) + δ(Σti-K) (decode) + ΣTi (tool latency)
  • Reduces to Tcached when α=0 (no speculation)
Key Insight
  • Optimal when Ti < δRi — tool finishes before reasoning phase ends, so result is ready for O2 & O3.
  • Best speedup at Ti ≈ δRi — masks entire tool call + saves most decode steps.
  • SOTA reasoning models have seconds-to-minutes reasoning traces — many common tools (ls, web search, file read) fall in this window.

Proposed Tool Cache API

POST /cache-tool-output/{response_id}

A simple API endpoint for providers to adopt engine-side speculation. Clients post speculated tool outputs; engine uses them to keep sequences resident.

// POST /cache-tool-output/{response_id} // Request body: JSON array of tool cache entries [ { "name": "web_search", // required - tool name as cache key "params": { // optional - canonicalized arguments "query": "latest CUDA version" }, "output": "CUDA 12.9 released...", // required - serialized tool output "keep_alive": 30 // optional - TTL in seconds } ] // Response 200 OK { "cached": 1 // number of entries accepted }
Provider Incentives
  • Reduces inference time per request → higher throughput
  • Fewer eviction/re-prefill cycles → better GPU utilization
  • Fits existing prefix-cache discount pricing models
User Incentives
  • Faster agent turnaround (10-20% time saved)
  • Reduced costs via cache-hit discounts
  • Simple integration: one HTTP POST per tool result

Experimental Setup

Hardware & Models
  • Hardware: Single node, 4× NVIDIA A100 80GB (NVLink), AMD EPYC 7763 64-core CPU
  • Main model M: gpt-oss-120b (SOTA reasoning model) on 1 A100 via vLLM
  • Speculative model S: xLAM (1B / 3B / 8B) — Llama fine-tuned on tool datasets. 3 data-parallel instances across remaining 3 A100s
  • xLAM-8B accuracy: ~80% tool prediction on BFCL
  • Commercial: Subset of experiments with gpt-5 + gpt-5-nano
Benchmark & Workload
  • Dataset: BFCL (Berkeley Function Calling Leaderboard) — diverse tool calling tasks with structured schemas
  • Concurrent agents: M ∈ {1, 8, 32} async agents
  • Each agent: 32 tasks, looping tool calls until final response
  • Tool latencies swept:
    • Short: mean ∈ {0, 0.1, ..., 0.5}s
    • Long: mean ∈ {0, 0.5, ..., 3.0}s
  • Speculation configs: λ ∈ {1, 3, 5, 7, 9} samples
  • 5 runs per config for statistical confidence

Results: Client-side Speculation

Throughput vs. Acceptance Rate
Throughput improvement with various speculative models. Higher acceptance rate → higher throughput. xLAM-8B reaches ~80%; smaller models improve with more samples (λ).
Time Saved vs. Tool Latency (32 agents)
6-21% end-to-end time saved per agent. Peak savings at tool latency ≈ generation time (~2-2.5s). Savings decrease for very long tools.

Client-side takeaway: Without any engine modifications, agents can save 6-21% time. xLAM-1B with 9× samples approaches xLAM-8B with 1× sample.

Results: Cost Analysis (gpt-5 + gpt-5-nano)

Commercial Model Viability
  • Client-side speculation works with any commercial API — tested with gpt-5 (main) + gpt-5-nano (speculative).
  • With 1 speculation per turn: ~10% time saved for only ~4% additional cost — the nano API call is an order of magnitude cheaper.
  • With 9 speculations: marginal time improvement, but cost rises to 25-30% of main generation cost.
  • Sweet spot: 1-3 speculations provide the best cost-benefit ratio.

Practical implication: A single cheap speculative call provides nearly all the benefit. Additional samples have diminishing returns relative to cost.

Cost-benefit analysis: % time saved vs. added cost for gpt-5 + gpt-5-nano. ~10% time savings for only ~4% price increase at λ=1.

Results: Engine-side Speculation

Additional Gains from Engine Integration
  • Engine-side algorithm provides 2-3% additional time savings on top of client-side speculation.
  • Best for short tools (0-1s): tool results arrive before main model finishes reasoning → O2 (early exit) and O3 (avoid eviction) both kick in.
  • Long tools (>1s): results arrive too late for engine-side benefits → equivalent to client-side only.
  • Current limitation: vLLM speculative decoding has high overhead at batch size > 1 (experimental phase). Results shown for 1 async agent.

Many common agent tools (ls, web search, file read, code lint) are in the 0-1s range — ideal for engine-side optimization.

Client-side vs. engine-side time savings for various tool latencies (1 agent, xLAM-8B). Engine-side adds 2-3% for short-latency tools.

Discussion & Limitations

Strengths
  • Client-side works with any API — no modifications needed; immediately deployable with commercial models.
  • Solid theoretical grounding: analytical model for speedup bounds, conditions for gains clearly stated.
  • Practically effective: 6-21% time saved for ~4% extra cost is a strong cost-benefit ratio for production agents.
  • Composable with existing optimizations: works alongside parallel tool calls, prefix caching, speculative decoding.
Limitations
  • Stateless tools only: tools with side effects (state modification, writes) cannot be safely speculated without rollback mechanisms.
  • Theoretical 2× bound: client-side approach can never exceed 2× speedup — can only hide one of generation or tool execution.
  • Engine-side batch>1 overhead: vLLM's speculative decoding at batch>1 is still experimental; limits multi-agent engine-side evaluation.
  • Tool prediction accuracy ceiling: speculation quality depends on speculative model's tool calling ability — niche/custom tools may have low hit rates.
Discussion Points
  • Connection to speculative decoding research: adapts the verify-then-accept paradigm from token-level to tool-level. The engine-side approach uses standard speculative sampling for validation, ensuring correctness.
  • Composability with parallel tool calling: when models emit multiple independent tool calls, each can be independently speculated — the approaches are orthogonal and stack.
  • Real-world generalization: BFCL provides diverse tools, but real-world agents may have highly custom tools with lower prediction accuracy. Domain-specific fine-tuning of speculative models could help.
  • Stateful tool speculation: future work could explore rollback/undo mechanisms for stateful tools, or transaction-like semantics for speculative execution.

Summary & Key Takeaways

Key Takeaways
  • Problem: Tool calls fragment LM agent inference, introducing sequential latency and eviction overhead that accumulate over multi-step tasks.
  • Solution: Speculative tool execution — use a small, fast model to predict tool calls and pre-execute them in parallel with main generation.
  • Two complementary approaches: Client-side (works with any API, 6-21% time saved) + Engine-side (additional 2-3% for short tools).
  • Theoretical guarantee: Sspec < 2, with best gains when tool latency T ≈ generation time G.
  • Practical impact: Up to 196 tok/s improvement; only ~4% extra cost with commercial APIs.
  • Proposed tool cache API enables easy adoption by inference providers.
