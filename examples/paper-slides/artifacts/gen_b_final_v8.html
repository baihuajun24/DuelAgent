The Problem / 问题在哪里
  • Tools dominate latency.Agent 调工具后，推理被切成多轮"生成 - 等待 - 再生成"。
  • Eviction hurts throughput.请求离开 batch；工具结果回来后还要重新调度和 re-prefill。
  • Prior fixes only help partially.并行工具只适合独立调用；prefix cache 省部分 prefill，但挡不住 eviction。
This Paper's Insight / 本文核心想法
Predict the entire tool call early, then launch it async.不是猜几个 token，而是直接猜完整 tool call，提前把工具跑起来。

Client-side Method: Speculate and Overlap / 客户端方法：投机并重叠等待

1
Parallel dispatch / 并行发起
Send the prompt to main model M and small spec model S together.
2
Speculate & launch / 预测并启动
If S predicts the tool first, start it async and cache the future.
3
Validate & reuse / 校验并复用
When M returns, hit means reuse; miss falls back to normal execution.

Takeaway: predict early, launch async, then reuse on a hit.纯客户端方案；今天就能部署。λ 表示并行猜测次数。

Theory Upper Bound: Intuition and Formula / 理论上界：直觉与公式

Speedup formula: depends on α (hit rate), g/G (spec-to-main ratio), and T relative to G. Higher α → more speedup; bound strictly < 2×.命中率越高越快，结构性上界不到 2 倍；客户端 speculation 只能 overlap 两段中的一段，最适合 T ≈ G 的场景。
Why ≤ 2×? / 为什么 < 2 倍？
  • Two phases: G + T per turn.每轮 = 生成 + 工具等待。
  • Overlap hides one, not both.最多消一段，严格 < 2×。
  • Sweet spot: T ≈ G.T 和 G 同量级时最优。
Speedup peaks near T ≈ G — bright region shows high speedup, concentrated where T≈G.

Engine-side Method I: Three Optimizations / 引擎侧方法 I：三个系统优化

Three Optimizations / 三个系统优化点
O1 Speculate tools / 提前猜工具
Predict the tool via S, then post the result to the engine tool cache.
O2 Early exit decoding / 提前结束解码
If the tool returns early, validate once and skip part of the decode.
O3 Avoid eviction / 避免被逐出 batch
On a hit, append the result to KV cache directly. No eviction, no re-prefill.

Requires engine changes.这是系统接口方案，不是现成 API 技巧；收益是在 client-side 之上继续叠加。

O1 hides latency, O2 skips decode, O3 avoids eviction三步都围绕同一件事：减少工具调用对推理链路的打断

Engine-side Method II: Tool Cache API / 引擎侧方法 II：Tool Cache API

Intuition / 关键直觉
  • Best case: the tool returns while the model is still reasoning.这样引擎就能直接校验并注入结果。
  • Condition: Ti < δRi.工具必须足够快，快到能赶在 reasoning 结束前回来。
Speculative Runtime (K turns)

Simple endpoint: one POST per speculated result.接口很薄；服务端能提吞吐，用户侧能降延迟。

Experiments III: Client-side Time Saved / 实验 III：客户端省时效果

  • 6-21% end-to-end time saved.只做 client-side，就已经有明显收益。
  • Peak at tool latency about 2-2.5 s.和前面的 T ≈ G 理论吻合。

No engine changes needed.这页是全文最有现实价值的结果。

6–21% time saved. Peak savings at T ≈ G (~2–2.5 s), exactly as theory predicts.

Experiments IV: Cost / 实验 IV：成本收益

  • About 10% faster for about 4% more cost.即使只有商业 API，这个方法也有现实价值。
  • Sweet spot: λ = 1-3.一次便宜猜测基本就拿走大部分收益。

One cheap speculative call provides nearly all the benefit.这里讲的是 black-box API 场景，不是前面 open-model 实验的横向对比。

~10% time savings for ~4% price increase at λ=1. Diminishing returns past λ=3.

Experiments V: Engine-side Gains / 实验 V：引擎侧额外收益

  • Extra 2-3% for short tools.在 client-side 基础上继续省一点，但前提是工具很快。
  • No gain for long tools.如果结果回来太晚，就赶不上 early exit。

Common tools (ls, search, file read) are 0-1 s.所以 engine-side 更适合快工具，而不是慢 API。

Engine-side adds 2–3% for tools under 1 s. Gains vanish as tool latency exceeds reasoning time.

Experiments VI: Main Takeaways / 实验 VI：结果小结

  • Deployable today.客户端方案不改 API，就能落地。
  • Theory matches experiments.上界和 sweet spot 都能被实验验证。
  • Practical even for paid APIs.商业模型下也有不错的时间 / 成本比。

Limitations / 局限

  • Mostly stateless tools.有副作用的工具很难安全 speculate。
  • Upper bound below 2x.这是方法本身的结构限制。
  • Engine-side evaluation is still limited.当前 vLLM 实现对 batch > 1 还不够成熟。

Summary / 总结

Speculative tool execution hides tool latency for LM agents.
把 speculative decoding 提升到工具层面，去抓住 agent 真正的延迟来源
  • Client-side: works today and saves real time.任何 API 都能用，省时大约 6-21%。
  • Engine-side: adds a bit more when tools are fast.额外收益不大，但系统方向是对的。
  • Impact: up to +196 tok/s with a simple tool-cache API.服务端和用户侧都能受益。

Backup: Speedup Proof & Runtime Assumptions

Lemma Proof Sketch
  • Monotonicity: M = max{G, g+T} ≤ G+T → denominator decreases in α → Sspec increases. S(0)=1 ⇒ S > 1 for α > 0.
  • Maximum: Smax = (G+T) / max{G, g+T} at α = 1.
  • Bound: max{a,b} ≥ (a+b)/2 gives Smax < 2 (strict, g > 0).
Runtime Assumptions
  • Previous-turn reasoning tokens not re-prefilled (standard for commercial APIs).
  • Prompt grows by tool call + output tokens each turn.
  • Tool outputs pre-computed; validated with real executions — no change in results.
